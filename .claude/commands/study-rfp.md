docs/00_srs/rfp-final.md (또는 복붙 .md)를 읽고, 사용자가 프로젝트 도메인을 체계적으로 학습할 수 있는 커리큘럼 + 강의자료를 생성한다.
design-orchestrator가 작업하는 동안 사용자가 병렬로 공부할 수 있도록 별도 폴더에 생성.

$ARGUMENTS 가 있으면 해당 레벨/도메인만 생성한다. (예: "Level 1만", "회원 도메인 딥다이브")

---

## 실행 전 필수 확인
1. docs/00_srs/ 에 rfp-final.md 또는 복붙 .md 존재 확인
2. docs/_meta/glossary.md 읽기 (이미 정리된 용어 참조)

---

## Step 1: 커리큘럼 설계 (순차)

rfp-final.md를 분석하여:
- 도메인 수, 프로세스 수, 복잡도 파악
- 학습 레벨과 순서 설계
- docs/00_srs/study/curriculum.md 생성

### curriculum.md 구조:
```markdown
# 학습 커리큘럼
> 프로젝트: {프로젝트명}
> 총 학습 시간: 약 {N}시간
> 도메인: {N}개 | 프로세스: {N}개

## Level 1: 전체 그림 (30분)
- 00-overview.md — 프로젝트 한눈에 보기
- 01-organization.md — 누가 쓰는 시스템인가

## Level 2: 도메인 기초 (도메인당 20분)
- 02-domain-map.md — 전체 도메인 지도
- 02-{DOMAIN}.md — 도메인별 개념 (도메인 수만큼)

## Level 3: 프로세스 이해 (프로세스당 15분)
- 03-process-map.md — 전체 프로세스 지도
- 03-{DOMAIN}-process.md — 도메인별 프로세스 (도메인 수만큼)

## Level 4: 업무규칙 + 엣지케이스 (도메인당 15분)
- 04-{DOMAIN}-rules.md — 도메인별 규칙 + 예외 시나리오

## Level 5: 종합 테스트 (30분)
- 05-final-test.md — 전체 이해도 확인 퀴즈

학습 경로: Level 1 → 2 → 3 → 4 → 5 (순서대로)
```

**사용자에게 커리큘럼 보여주고 승인 요청** → 승인 후 강의자료 생성

---

## Step 2: Level 1 — 전체 그림 (🔀 병렬 2개)

**에이전트 A: 프로젝트 오버뷰**
- 출력: `docs/00_srs/study/00-overview.md`
- 내용:
  - **한 줄 요약**: 이 프로젝트가 뭔지 한 문장
  - **5분 브리핑**: 프로젝트 전체를 5분 안에 설명
  - 전체 시스템 조감도 (Mermaid)
  - 핵심 수치 (기능 수, 화면 수, 사용자 수, 도메인 수)
  - 이 프로젝트에서 **가장 중요한 3가지**
  - 이 프로젝트에서 **가장 복잡한 3가지**
  - **퀴즈 3문항**

**에이전트 B: 조직/이해관계자**
- 출력: `docs/00_srs/study/01-organization.md`
- 내용:
  - 발주 조직 분석 (어떤 업종, 어떤 규모)
  - 사용자 유형별 페르소나 (이름, 역할, 하루 일과, 시스템 사용 목적)
  - 역할별 권한 범위 (뭘 볼 수 있고, 뭘 할 수 있는지)
  - 조직도 (Mermaid)
  - **퀴즈 3문항**

---

## Step 3: Level 2 — 도메인 기초 (🔀 대규모 병렬 — 도메인당 1에이전트)

**에이전트 Z: 도메인 지도** (먼저 1개)
- 출력: `docs/00_srs/study/02-domain-map.md`
- 내용:
  - 전체 도메인 관계도 (Mermaid)
  - 도메인 간 데이터 흐름
  - 도메인 우선순위/의존 순서
  - "어디서부터 봐야 하나?" 추천 순서

**에이전트 ×N: 도메인별 딥다이브** (도메인 지도 완료 후 병렬)
- 출력: `docs/00_srs/study/02-{DOMAIN}.md` (도메인당 1파일)
- 각 에이전트가 읽을 것: rfp-final.md (해당 도메인 부분), glossary.md
- 내용:
  - **이 도메인이 뭔가** — 비유로 설명 ("이건 마치 ~와 같다")
  - **핵심 개념 5개** — 각각 예시와 함께
  - **이 도메인의 데이터** — 어떤 정보를 다루는가 (표)
  - **다른 도메인과의 관계** — "주문 도메인은 회원 도메인에서 사용자 정보를 받아..."
  - **실생활 비유** — 도메인 전체를 일상 경험에 비유
  - **업계 배경지식** — 이 도메인을 이해하려면 알아야 할 업계 상식
  - **자주 하는 실수** — 이 도메인에서 개발자가 흔히 놓치는 것
  - **퀴즈 5문항** (개념 이해 + 관계 이해)

---

## Step 4: Level 3 — 프로세스 이해 (🔀 도메인별 병렬)

**에이전트 Y: 프로세스 지도** (먼저 1개)
- 출력: `docs/00_srs/study/03-process-map.md`
- 내용:
  - 전체 프로세스 관계도 (어떤 프로세스가 어떤 프로세스를 트리거하는지)
  - 프로세스 복잡도 순위 (가장 복잡한 것부터)
  - 프로세스 간 공유 데이터

**에이전트 ×N: 도메인별 프로세스** (프로세스 지도 완료 후 병렬)
- 출력: `docs/00_srs/study/03-{DOMAIN}-process.md` (도메인당 1파일)
- 내용:
  - 각 프로세스를 **사용자 스토리**로 설명:
    ```
    📖 시나리오: 신규 주문 등록

    김대리(영업팀)는 오전 10시에 고객사로부터 전화를 받았다.
    "A 부품 500개를 다음 주 금요일까지 납품해주세요."

    김대리는 시스템에 로그인하여...
    1. [주문 등록] 화면에서 고객사를 검색한다
    2. 부품 A를 선택하고 수량 500을 입력한다
    3. 납기일을 다음 주 금요일로 설정한다
    4. [저장] 버튼을 클릭한다
       → 시스템이 재고를 확인한다
       → 재고 부족 시: "재고가 300개입니다. 계속하시겠습니까?"
       → 김대리가 [확인]을 누르면 부족분 200개 자동 발주 요청
    5. 주문이 등록되고, 생산팀 박과장에게 알림이 간다
    ```
  - 각 프로세스 흐름도 (Mermaid flowchart)
  - **분기점 상세**: 조건 → 결과 표
  - **예외 시나리오**: "만약 ~하면?" 3가지 이상
  - **퀴즈 5문항** (흐름 이해 + 예외 대응)

---

## Step 5: Level 4 — 업무규칙 + 엣지케이스 (🔀 도메인별 병렬)

- 출력: `docs/00_srs/study/04-{DOMAIN}-rules.md` (도메인당 1파일)
- 내용:
  - 이 도메인의 **절대 위반 불가 규칙** (표)
    | 규칙 | 왜 존재하나 | 위반하면 어떻게 되나 | 실수 예시 |
  - **경계값/엣지케이스 시나리오**:
    - "수량이 0이면?", "날짜가 과거이면?", "동시에 두 명이 같은 걸 수정하면?"
  - **도메인 간 충돌 시나리오**:
    - "주문은 확정됐는데 재고가 없으면?"
    - "승인자가 퇴사했으면?"
  - 각 시나리오의 **정답** (시스템이 어떻게 처리해야 하는지)
  - **퀴즈 5문항** (함정 포함)

---

## Step 6: Level 5 — 종합 테스트 (순차)

- 출력: `docs/00_srs/study/05-final-test.md`
- 내용:
  - **Part A: 개념 확인** (10문항) — 도메인, 용어, 조직
  - **Part B: 프로세스 추적** (5문항) — "이 시나리오에서 다음 단계는?"
  - **Part C: 예외 대응** (5문항) — "이 상황에서 시스템은 어떻게 해야 하나?"
  - **Part D: 통합 시나리오** (3문항) — 여러 도메인을 걸치는 복합 상황
  - 각 문항에 정답 + 해설 (접어두기로 숨김)
  - **채점 기준**: 80% 이상이면 "프로젝트 이해 충분", 미만이면 "복습 필요 도메인 안내"
