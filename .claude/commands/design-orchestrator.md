## 핵심 철학: 병렬 에이전트 실행

> 문서를 나누는 이유는 **서브에이전트가 병렬로 작업하기 위함**이다.
> 1파일 = 1에이전트가 담당할 수 있는 단위.
> 독립적인 작업은 반드시 서브에이전트로 동시 실행한다.

### 병렬 실행 원칙
- **작성 병렬화**: 도메인별로 분리된 파일은 서브에이전트가 동시에 작성
- **검증 병렬화**: 문서별 검증도 서브에이전트가 동시에 수���
- **동기화는 병렬 작업 완료 후**: 모든 서브에이전트 결과를 모아서 교차 검증
- **의존성 있는 작업만 순차**: Phase 간은 순차, Phase 내 독립 Step은 병렬

## Phase 3: IA (도메인 구조 + 메뉴 설계)
> ⚡ 순차 실행 — 이후 모든 Phase의 기반이 되는 구조 확정

### Step 3.1: 도메인 구조 정의
- 각 도메인이 **왜** 존재하는가 (목적, 책임)
- 도메인 간 **경계**: 어디까지가 이 도메인의 범위인가
- 도메인 간 **관계**: 의존성, 데이터 흐름 방향
- 도메인 **소유 엔티티**: 이 도메인이 관리하는 핵심 데이터

### Step 3.2: 메뉴 구조 설계
- 각 메뉴/페이지가 **무엇을 위해** 있는가
- 메뉴 ↔ 기능 매핑 (F-ID 참조)
- 메뉴 접근 권한 조건 (역할별)
- 메뉴 계층 구조 (Mermaid 또는 트리)

### Step 3.3: 네비게이션 플로우
- 메뉴 간 이동 경로
- 진입점 (랜딩 페이지, 딥링크)
- 이동 시 필요한 전제조건 (로그인, 권한, 이전 단계 완료 등)

### Step 3.4: 문서 생성
- docs/02_analysis/ia.md 생성

### Phase 3 리포트
```
✅ 도메인: {N}개 확정
✅ 메뉴: {N}개 (1depth: {n} / 2depth: {n})
✅ 네비게이션 플로우: {N}개 경로
📄 생성: ia.md
⚠️ 확인 필요: 도메인 경계와 메뉴 구조가 맞는지
→ 승인하시겠습니까?
```

## Phase 4: 프로세스 + 업무규칙
> ⚡ 병렬: IA 확정 후, 프로세스와 업무규칙 동시 작성
> ia.md를 공통 기반으로 참조

### Step 4.1 ~ 4.2 (🔀 병렬 — 서브에이전트 2그룹)

**에이전트 그룹 A: 프로세스 (도메인별 병렬 가능)**
- ia.md + glossary.md + feature-definition 참조
- 도메인별 주요 업무 플로우 (사용자 시나리오)
- 각 프로세스:
  - 트리거 (시작 조건)
  - 단계별 흐름 (정상/예외)
  - 제약조건: 순서 의존성, 데이터 정합성, 동시성 제약
  - 관련 기능 ID, 관련 화면(있으면)
- docs/02_analysis/process.md (또는 도메인별 분리)
- 프로세스 20개 초과 시 도메인별: process-{DOMAIN}.md

**에이전트 그룹 B: 업무규칙 (도메인별 병렬 가능)**
- ia.md + glossary.md + feature-definition 참조
- 도메인별 비즈니스 룰 (위반 불가 규칙)
- 각 규칙:
  - 규칙 ID (BR-{DOMAIN}-{NNN})
  - 규칙 내용 (명확한 조건 → 결과)
  - 적용 범위 (어떤 기능/프로세스에 적용)
  - 예외 조건 (있으면)
  - 위반 시 동작 (에러 메시지, 차단, 경고 등)
- docs/02_analysis/business-rules.md (또는 도메인별 분리)
- 규칙 30개 초과 시 도메인별: business-rules-{DOMAIN}.md

### Step 4.3: 교차 검증 (병렬 완료 후)
- 프로세스 ↔ 업무규칙 정합성
  - 프로세스의 제약조건이 업무규칙에 정의되어 있는가?
  - 업무규칙이 실제 프로세스에서 적용되는가?
- IA 도메인 ↔ 프로세스 도메인 ↔ 업무규칙 도메인 일치 여부

### Phase 4 리포트

```
🔀 병렬 실행: 프로세스 {n}개 에이전트 + 업무규칙 {n}개 에이전트
✅ 교차검증: 프로세스↔규칙 {N}/{N} 매핑 확인
```

- **서브에이전트 N개 동시 실행** (화면당 1개)
- 각 서브에이전트가 읽을 공통 컨텍스트:

- 각 SCR-NNN.md에 포함할 내용:

  **1. 화면 목적**
  - 왜 존재하나, 사용자가 뭘 달성하려는가
  - 어떤 프로세스의 어떤 단계인가 (process.md 참조)
  - 진입 조건 (이전 화면, 권한, 상태)

  **2. 와이어프레임**
  - ASCII 또는 Mermaid로 레이아웃
  - 영역별 역할 표시 (헤더, 사이드바, 컨텐츠, 푸터)
  - design-system.md 컴포넌트 사용

  **3. 상세 기능 목록**
  - 화면 내 모든 요소 (표로 작성):
    | 요소 | 타입 | 기본값 | 필수 | 설명 |
  - 입력 필드: 데이터타입, 최대길이, 포맷, 플레이스홀더
  - 버튼: 레이블, 활성화 조건
  - 테이블: 컬럼 구성, 정렬, 페이징
  - 필터/검색: 검색 대상 필드, 조건 조합

  **4. 상호작용 (Interaction)**
  - 이벤트 → 동작 매핑 (표로 작성):
    | 트리거 | 대상 | 동작 | 결과 |
  - 클릭, 호버, 포커스, 키보드, 스크롤, 드래그
  - 실시간 반응: 입력 중 유효성, 자동완성, 디바운스

  **5. 화면 내 프로세스**
  - 핵심 버튼별 조건 분기:
    ```
    [저장 버튼 클릭]
    → 유효성 검사 통과?
      → Yes: API 호출 → 로딩 ��� 성공/실패 분기
      → No: 에러 필드 하이라이트 + 메시지 표시
    ```
  - 상태 전이: 초기 → 입력중 → 제출 → 로딩 → 성공/실패
  - 조건부 표시/숨김: "권한이 ADMIN이면 삭제 버튼 노출"
  - 업무규칙 적용 시점: 어떤 BR이 이 화면에서 체크되는지

  **6. API 연동**
  - 호출 시점 (페이지 로드, 버튼 클릭, 스크롤 등)
  - api-spec.md 참조 (엔드포인트, 요청/응답)
  - 로딩 상태 UI
  - 에러 응답별 화면 반응:
    | 에러코드 | 화면 동작 |
  - 낙관적 업데이트 여부

  **7. 엣지 케이스**
  - 빈 데이터 (목록 0건, 검색 결과 없음)
  - 권한 없음 (접근 차단 vs 기능 숨김 vs 비활성화)
  - 네트워크 에러 (재시도 UI, 오프라인 처리)
  - 대량 데이터 (무한스크롤, 가상화, 로딩 전략)
  - 동시 편집 (충돌 시 처리)
  - 브라우저 뒤로가기/새로고침 시 상태 보존

## Phase 9: 동기화 + 검증
> ⚡ 검증도 병렬 — 매핑별 서브에이전트가 동시에 체크

### Step 9.1: 추적성 검증 (🔀 병렬)
- **서브에이전트 N개가 동시에 검증:**
  - 에이전트 A: feature ↔ screen 매핑 (모든 기능에 화면이 있는가)
  - 에이전트 B: feature ↔ API 매핑 (모든 기능에 API가 있는가)
  - 에이전트 C: feature ↔ TC 매핑 (모든 기능에 TC가 있는가)
  - 에이전트 D: API ↔ error-code 매핑 (모든 API 에러가 정의되었는가)
  - 에이전트 E: screen ↔ API 매핑 (화면의 API 연동이 맞는가)
  - 에이전트 F: process ↔ screen 매핑 (프로세스 단계별 화면이 있는가)
  - 에이전트 G: business-rules ↔ screen/API 적용 확인
- 각 에이전트가 누락/불일치 목록 반환

### Step 9.2: 용어 일관성 검증 (🔀 병렬)
- **서브에이전트가 문서별로 glossary 대조**
- glossary에 없는 도메인 용어 탐지

### Step 9.3: 품질 검증 — "읽고 바로 코딩 가능한가?"
- 문서별 체크리스트:
  | 검증 항목 | 체크 |
